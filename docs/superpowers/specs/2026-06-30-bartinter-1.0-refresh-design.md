# Bartinter 1.0 — Refresh Design

**Date:** 2026-06-30
**Status:** Approved design → implementation plan pending
**Author:** Maxim Kotliar (with Claude)

## 1. Summary

Bartinter dynamically sets the iOS status bar style (light vs dark content) based on
the brightness of the content behind it. The library is popular (≈1,700 stars) but the
implementation is from 2018: Swift 5.0, iOS 9 floor, CocoaPods-only, two global
Objective-C swizzles, a deprecated graphics-context capture path, and a `.default`/
`.lightContent` decision that breaks under Dark Mode.

Version 1.0 is a clean major rewrite that keeps the product promise (dead-simple,
"it just works" status-bar tinting) while modernizing the entire internal pipeline and
adding first-class SwiftUI support.

### Decisions locked during brainstorming

| Decision | Choice |
| --- | --- |
| Minimum iOS | **iOS 15** |
| SwiftUI | **First-class SwiftUI API**, plus the UIKit path |
| Compatibility | **New major version (1.0), free to redesign the public API** |
| Distribution | **Swift Package Manager only** (drop CocoaPods / Carthage) |
| Injection (no swizzling) | **Hybrid:** app-wide window/scene install + per-screen control |
| Capture pipeline | **Hybrid:** `CARenderer` → `MTLTexture` primary on device + automatic `layer.render(in:)` CPU fallback (CARenderer is a no-op on the iOS simulator). Metal averaging in both. |

## 2. Goals / Non-Goals

### Goals
- Remove **both** global swizzles (`UIViewController.childForStatusBarStyle`,
  `UIView.layoutSubviews`).
- Replace the deprecated `UIGraphicsBeginImageContext` + `UIApplication.statusBarFrame`
  capture path with a modern GPU pipeline.
- Fix Dark Mode by deciding strictly between `.lightContent` and `.darkContent`.
- Stop the notch / Dynamic Island from skewing the brightness average.
- Stop spurious changes from scroll bounce / unrelated layout.
- Ship a proper `Package.swift` (SPM is currently broken — `Package.swift` is absent
  despite the merged PR #14).
- Provide a first-class SwiftUI modifier and a clean UIKit API.
- Swift 6 language mode, strict concurrency.

### Non-Goals
- Capturing arbitrary on-screen pixels (impossible without ReplayKit; out of scope).
- Correctly sampling *live server-composited* content under the bar (active
  `UIVisualEffectView` blur, `AVPlayerLayer`) — see §8 Known Limitations.
- Backward compatibility with the 0.0.x public API (clean break; migration guide provided).
- macOS / tvOS / watchOS support.

## 4. Architecture

Status bar style must be supplied by a `UIViewController` via
`preferredStatusBarStyle` — there is no public SwiftUI status-bar-*style* API even on
iOS 18 (verified). So a lightweight controller remains the style provider, but it no
longer relies on swizzling and no longer drives sampling off global layout.

```
                 ┌─────────────────────────────────────────────┐
                 │ BartinterController (UIViewController)        │
                 │  - owns preferredStatusBarStyle               │
                 │  - injected via childForStatusBarStyle        │
                 └───────────────┬─────────────────────────────┘
                                 │ requests a style
                 ┌───────────────▼─────────────────────────────┐
                 │ TintEngine                                    │
                 │  - orchestrates sample → average → decide     │
                 └───┬───────────────┬───────────────┬──────────┘
            triggers │        capture │       average │
        ┌────────────▼──┐   ┌─────────▼────┐   ┌──────▼─────────┐
        │ SampleScheduler│   │ RegionCapture│   │ LuminanceMath  │
        │ (CADisplayLink │   │ (CARenderer →│   │ (linearize +   │
        │  + dirty flag) │   │  MTLTexture) │   │  Rec.709 +     │
        └────────────────┘   └──────────────┘   │  hysteresis)   │
                                                 └────────────────┘
```

### Modules (each independently testable)

1. **`BartinterController`** — `UIViewController` subclass. Only responsibility: hold the
   current `UIStatusBarStyle` and override `preferredStatusBarStyle` /
   `preferredStatusBarUpdateAnimation`; animate `setNeedsStatusBarAppearanceUpdate()` on
   change. No capture logic.
2. **`TintEngine`** — `@MainActor` orchestrator. Holds configuration, wires the scheduler
   to capture → average → decision, applies hysteresis, and pushes the resulting style to
   the controller. The unit that "decides".
3. **`SampleScheduler`** — replaces the `layoutSubviews` swizzle. A `CADisplayLink`-driven
   sampler that runs only while the host is visible and a **dirty** flag is set, throttled
   to `maxSampleRate`, and **auto-idles** after the luminance is stable for N frames.
   Dirty is set by: `UIScrollView.contentOffset` KVO (when a scroll view is registered),
   explicit `setNeedsStatusBarTintUpdate()`, view appearance, and trait/size changes.
4. **`RegionCapture`** — GPU-native capture. Computes the sampling regions (§4.2), renders
   the host layer tree into a small private `MTLTexture` via `CARenderer`, and hands the
   texture to averaging. Owns a long-lived `MTLDevice`/`CARenderer`/`CIContext`.
5. **`LuminanceMath`** — pure functions: sRGB→linear, Rec.709 relative luminance, and the
   light/dark decision with anti-flicker hysteresis. Zero dependencies; fully unit-tested.
6. **`StatusBarGeometry`** — derives the capture rects, including the notch/Dynamic Island
   exclusion (§4.2), from `windowScene.statusBarManager?.statusBarFrame` and
   `safeAreaInsets`.

### 4.1 Capture & averaging pipeline (hybrid: GPU primary, CPU fallback)

1. `StatusBarGeometry` produces 1–2 source rects in the host view's coordinate space
   (left + right readable regions, excluding the center band).
2. `RegionCapture` configures a `CARenderer(mtlTexture:options:)` (iOS 11+) targeting a
   small private `MTLTexture` (e.g. downscaled to ~16×N — averaging tolerates heavy
   downsampling, so cost is dominated by setup, not pixels), assigns the host
   `view.layer`, and renders the requested rect(s).
   **Automatic CPU fallback:** `CARenderer` is a no-op on the iOS simulator (produces an
   all-zero texture). `RegionCapture` detects this with a pixel probe and falls back to
   `layer.render(in:)` via `UIGraphicsImageRenderer` for that rect, so integration tests
   pass on simulator and the production on-device path is preserved unchanged.
3. Averaging runs via `CIAreaAverage` on a **long-lived Metal-backed `CIContext`**
   (or the default CIContext on simulator), reducing to ~1×1.
4. The single resulting pixel is read back **asynchronously** (background queue — never a
   synchronous main-thread GPU wait at sample rate).
5. `LuminanceMath` linearizes (WCAG sRGB transfer) and applies Rec.709 weights
   `0.2126 / 0.7152 / 0.0722`. Alternatively request linear-sRGB readback and weight
   directly.

The Metal device, texture, and CIContext are created once and reused across samples.

### 4.2 Notch / Dynamic Island handling

There is no public API for the Dynamic Island / notch rect (verified). Strategy:
- If `safeAreaInsets.top` indicates a notch/island device, compute a **center exclusion
  band** as a fraction of width centered horizontally, and sample only the **left** and
  **right** readable regions, area-weighting their averages.
- On non-notch devices, sample the full status-bar strip.
- This keeps the black island/notch from dragging the average toward "dark" and is
  derived purely from public geometry (no per-device hardcoding).

### 4.3 Style decision & anti-flicker

- Decide strictly between `.lightContent` and `.darkContent`. **Never `.default`** — it
  adapts to the system Dark Mode setting and is the root cause of #15.
- Keep the hysteresis band (`midPoint` ± `antiFlickRange/2`) so values hovering near the
  threshold don't oscillate.

## 5. Public API (approved surface; exact names confirmed in the plan)

Prerequisite (unchanged): `UIViewControllerBasedStatusBarAppearance` must be `YES` in
Info.plist (the default).

### SwiftUI — the common case
SwiftUI has no public status-bar-*style* API, so `.tintsStatusBar()` bridges to the
window root for the integrator. The App-lifecycle case needs no separate install call —
the modifier reaches the window lazily on first appearance.

```swift
@main
struct GalleryApp: App {
    var body: some Scene {
        WindowGroup {
            ContentView()
                .tintsStatusBar()                 // samples content behind the bar
        }
    }
}

// Per-screen, with config / pause:
PhotoDetailView()
    .tintsStatusBar(Bartinter.Configuration(midPoint: 0.55, maxSampleRate: 30))

PlayerView()
    .tintsStatusBar(isActive: autoTint)           // false = stop sampling, keep last style
```

### UIKit — app-wide install (zero per-screen code)
```swift
func scene(_ scene: UIScene, willConnectTo session: UISceneSession,
           options: UIScene.ConnectionOptions) {
    guard let windowScene = scene as? UIWindowScene else { return }
    // ... build window / root ...
    Bartinter.install(in: windowScene)            // owns the window's status bar
}
```

### UIKit — per-screen control
```swift
final class GalleryViewController: UIViewController {
    private let bartinter = BartinterController()

    override var childForStatusBarStyle: UIViewController? { bartinter }

    override func viewDidLoad() {
        super.viewDidLoad()
        bartinter.tint(self)               // sample this VC's content behind the bar
        bartinter.observe(collectionView)  // explicit opt-in: re-sample on scroll (KVO)
    }
}

// Container forwarding (fixes #8):
final class TintingNavigationController: UINavigationController {
    override var childForStatusBarStyle: UIViewController? { topViewController }
}
```

> **Decision:** `observe(_:)` is an **explicit opt-in** in 1.0 (predictable; no
> hierarchy-walking). Auto-discovery of scroll views is a possible later enhancement.

### Manual control & state
```swift
bartinter.setNeedsStatusBarTintUpdate()          // force a re-sample now
bartinter.isActive = false                        // pause / resume sampling
let style = bartinter.currentStyle                // .lightContent / .darkContent
```

### Configuration
```swift
public struct Configuration {
    public var animationDuration: TimeInterval     // 0.2
    public var animationType: UIStatusBarAnimation // .fade
    public var midPoint: CGFloat                    // 0.6  — luminance threshold
    public var antiFlickRange: CGFloat              // 0.08 — hysteresis band width
    public var maxSampleRate: Double                // 12   — samples/sec ceiling (replaces throttleDelay)
}

// App-wide defaults set once at launch:
Bartinter.Configuration.default.midPoint = 0.6
```

## 6. Data flow

trigger (scroll KVO / explicit / appearance) → `SampleScheduler` marks dirty → next
display-link tick (throttled) → `RegionCapture` renders region(s) to texture → GPU
average → async readback → `LuminanceMath` decides style → `TintEngine` applies
hysteresis → `BartinterController.statusBarStyle` changes → animated
`setNeedsStatusBarAppearanceUpdate()`. Scheduler idles after N stable frames.

## 7. Error handling
- No Metal device (shouldn't happen on supported hardware) → log once, no-op (status bar
  keeps its current style); never crash.
- Missing `windowScene` / zero-size status bar frame → skip the sample, stay idle.
- Capture/readback failure → drop the sample; never block the main thread.

## 8. Known limitations (documented)
- Both the GPU path (`CARenderer`) and the CPU fallback (`layer.render(in:)`) render the
  layer **model** tree and therefore do **not** capture *live server-composited* content
  (active `UIVisualEffectView` blur, `AVPlayerLayer` video) under the status bar.
  Acceptable: status-bar-region content is almost always opaque backgrounds, images, or
  gradients. A `drawHierarchy`-based fallback could be added behind `capturesEffectContent`
  if demand appears; not in 1.0 scope.
- On the iOS simulator `CARenderer` produces no GPU output; the automatic `layer.render(in:)`
  CPU fallback ensures correct behavior for testing, with no API changes required from
  integrators.

## 9. Concurrency
- `BartinterController`, `TintEngine`, `SampleScheduler` are `@MainActor`.
- GPU readback happens off-main (completion handler / background queue); only the final
  style assignment hops back to the main actor.
- Swift 6 language mode, strict concurrency checking on.

## 11. Testing strategy
- **`LuminanceMath`** — unit tests: sRGB linearization, Rec.709 weighting, decision
  thresholds, hysteresis behavior (no oscillation across the band).
- **`StatusBarGeometry`** — unit tests: notch vs non-notch rect derivation, exclusion-band
  math, zero/invalid frame handling.
- **`SampleScheduler`** — unit tests: dirty/idle state machine, throttle ceiling, stops
  when not visible.
- **`RegionCapture`** — integration test on a Metal-capable simulator: render a known
  solid-color layer, assert the averaged luminance and resulting style.
- **Example app UI test** — light vs dark background flips the status bar style.

## 13. Performance budget

Estimates for a modern A-series device, 120 Hz, a moderately complex screen. Absolute
numbers need on-device Instruments verification; the relative picture is structural.

**Why 1.0 is a net win:** the old design has an invisible **app-wide tax** — the
`layoutSubviews` swizzle runs an associated-object lookup on *every* `UIView`'s layout
(~0.5–1.5% of a core during scroll, scaling with the host app's view count). 1.0 removes
it entirely (O(1)-per-frame `CADisplayLink` + one `contentOffset` KVO), and moves capture
from CPU `layer.render(in:)` to GPU `CARenderer`, where the compositor already has headroom.

| Dimension | Old (0.0.x) | New (1.0) |
| --- | --- | --- |
| App-wide swizzle tax | ~0.5–1.5% core during scroll | none |
| Main-thread CPU / sample | 2–15 ms (unsafe, off-main) | ~0.05–0.2 ms |
| GPU / sample | ~none (CPU path) | ~0.1–1 ms |
| Allocations / sample | new CIContext + 2 buffers each time | zero (reused) |
| Idle / static screen | still fires on layout (≤5 Hz) | true idle ≈ 0 |
| Active-scroll total | ~3–4% CPU core | ~0.3% CPU + ~1.5% GPU |

- **Idle:** ≈ 0% (sampler parks after the luminance stabilizes for N frames).
- **Active scroll:** ~0.3% CPU + ~1.5% GPU, and removal of the app-wide tax means most
  apps see a net CPU **reduction** despite a higher sample rate.
- **Readback** is async (command-buffer completion / background queue) → ~1-frame tint
  latency, never a main-thread GPU stall.
- **Default `maxSampleRate` = 12 Hz** — a status-bar tint does not need 30–120 Hz;
  conservative default halves per-second GPU cost vs 30 Hz. Exposed for tuning.
- **Risk to measure first:** `CARenderer` traversal cost scales with layer-tree
  complexity (same variable as `render(in:)`, just GPU). Clip the render to the
  status-bar region so offscreen geometry is culled.
